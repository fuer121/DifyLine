# 小说角色定位增量合并工作流设计

## 目标

基于输入的小说 ID 和章节范围，最多处理 100 章。工作流逐章获取章节内容，识别章节中出现的角色，判断角色定位为「核心角色」「重要角色」或「边缘角色」，并在每章分析后把结果与历史暂存结果增量合并、去重、更新定位和依据。最终输出完整角色清单。

最终输出字段：

```json
[
  {
    "角色名": "角色名称",
    "角色定位": "核心角色 | 重要角色 | 边缘角色",
    "判断依据": ["依据 1", "依据 2"]
  }
]
```

## 推荐节点结构

```text
用户输入
  -> 章节范围校验与初始化
  -> If/Else：范围是否合法
      -> 不合法：输出错误信息
      -> 合法：Loop：逐章分析并增量合并
          -> If/Else：cursor >= total_chapters
              -> 是：Exit Loop
              -> 否：取当前章节
                    -> 获取书库书籍章节内容
                    -> LLM：单章角色分析
                    -> 代码：解析单章 JSON
                    -> LLM：增量合并角色结果
                    -> 代码：规范化合并结果
                    -> 变量赋值器：更新 role_state、cursor
  -> 最终格式化输出
  -> 输出
```

### 2. 章节范围校验与初始化

节点类型：代码节点  
语言：Python 3

输入：

| 变量 | 来源 |
| --- | --- |
| `book_id` | 用户输入.book_id |
| `start_chapter_index` | 用户输入.start_chapter_index |
| `end_chapter_index` | 用户输入.end_chapter_index |

输出：

| 变量 | 类型 |
| --- | --- |
| `is_valid` | boolean |
| `error_message` | string |
| `chapter_numbers` | array[number] |
| `total_chapters` | number |
| `cursor` | number |
| `role_state` | string |

代码：

```python
def main(book_id: int, start_chapter_index: int, end_chapter_index: int):
    try:
        book_id = int(book_id)
        start = int(start_chapter_index)
        end = int(end_chapter_index)
    except Exception:
        return {
            "is_valid": False,
            "error_message": "book_id、start_chapter_index、end_chapter_index 必须是数字",
            "chapter_numbers": [],
            "total_chapters": 0,
            "cursor": 0,
            "role_state": "[]",
        }

    if book_id <= 0:
        return {
            "is_valid": False,
            "error_message": "book_id 必须大于 0",
            "chapter_numbers": [],
            "total_chapters": 0,
            "cursor": 0,
            "role_state": "[]",
        }

    if start <= 0 or end <= 0:
        return {
            "is_valid": False,
            "error_message": "章节 sortid 必须大于 0",
            "chapter_numbers": [],
            "total_chapters": 0,
            "cursor": 0,
            "role_state": "[]",
        }

    if end < start:
        end = start

    total = end - start + 1
    if total > 100:
        return {
            "is_valid": False,
            "error_message": "章节范围最多支持 100 章，请缩小 start_chapter_index 和 end_chapter_index 的范围",
            "chapter_numbers": [],
            "total_chapters": 0,
            "cursor": 0,
            "role_state": "[]",
        }

    chapter_numbers = list(range(start, end + 1))
    return {
        "is_valid": True,
        "error_message": "",
        "chapter_numbers": chapter_numbers,
        "total_chapters": total,
        "cursor": 0,
        "role_state": "[]",
    }
```

### 3. If/Else：范围是否合法

条件：

```

```text

### 4. Loop：逐章分析并增量合并

使用 Loop，而不是 Iteration。原因是本需求需要循环变量保存状态：每处理一章，都要继承上一章合并后的 `role_state`。

Loop 变量建议：

| 变量 | 初始值来源 | 类型 | 用途 |
| --- | --- | --- | --- |
| `cursor` | 章节范围校验与初始化.cursor | number | 当前处理到章节数组的下标 |
| `role_state` | 章节范围校验与初始化.role_state | string | 已合并的角色状态，JSON 字符串 |

Loop 最大次数：

```text
100
```

Loop 内第一个节点应判断：

```text
cursor >= total_chapters
```

若为 true，走 Exit Loop。

### 7. LLM：单章角色分析

模型可继续使用原工作流的 `deepseek-chat`。建议 temperature 调低到 `0.2` 或 `0.3`，因为这里需要稳定结构化抽取。

系统提示词：

```text
你是小说角色识别与角色定位分析专家。你需要根据单章小说内容，识别本章中实际出现或被明确提及的角色，并判断每个角色在全书叙事中的阶段性定位。

角色定位只能从以下三类选择：
1. 核心角色：主角、主要反派、主要同伴、主要对手、长期驱动主线冲突或主线成长的人物。
2. 重要角色：对本章或近期情节有明显推动作用，和核心角色存在明确关系，参与关键事件、冲突、任务、势力关系或重要信息传递的人物。
3. 边缘角色：短暂出场、背景人物、工具性人物、路人、只被轻微提及，或目前缺少证据证明其重要性的人物。

判断要求：
- 只根据章节原文判断，不得脑补后续剧情。
- 如果证据不足，优先判为较低定位。
- 同一个角色的不同称呼、身份称号、外号可以写入 aliases，但不要把缺乏证据的同姓、同职位、同称号人物强行合并。
- 判断依据必须来自原文，可概括但不能编造。
- 输出必须是合法 JSON，不要输出 Markdown，不要输出解释性文字。

输出 JSON 结构：
{
  "chapter_index": 章节编号,
  "chapter_title": "章节标题",
  "roles": [
    {
      "name": "标准角色名",
      "aliases": ["别名或称谓"],
      "position": "核心角色 | 重要角色 | 边缘角色",
      "evidence": [
        {
          "text": "原文依据或高度贴近原文的简短概括",
          "reason": "为什么该依据支持这个定位"
        }
      ]
    }
  ]
}
```

用户提示词：

```text
章节编号：{{#取当前章节.current_chapter#}}

章节内容：
{{#获取书库书籍章节内容.text#}}
```

### 8. 代码：解析单章 JSON

节点类型：代码节点  
语言：Python 3

输入：

| 变量 | 来源 |
| --- | --- |
| `text` | LLM：单章角色分析.text |
| `current_chapter` | 取当前章节.current_chapter |

输出：

| 变量 | 类型 |
| --- | --- |
| `chapter_roles_json` | string |

代码：

```python
def main(text: str, current_chapter: int):
    import json
    import re

    raw = (text or "").strip()
    raw = re.sub(r"^```(?:json)?", "", raw).strip()
    raw = re.sub(r"```$", "", raw).strip()

    try:
        data = json.loads(raw)
    except Exception:
        data = {
            "chapter_index": int(current_chapter),
            "chapter_title": "",
            "roles": [],
            "parse_error": True,
            "raw_text": raw[:1000],
        }

    if not isinstance(data, dict):
        data = {
            "chapter_index": int(current_chapter),
            "chapter_title": "",
            "roles": [],
            "parse_error": True,
            "raw_text": raw[:1000],
        }

    roles = data.get("roles")
    if not isinstance(roles, list):
        data["roles"] = []

    return {
        "chapter_roles_json": json.dumps(data, ensure_ascii=False),
    }
```

### 9. LLM：增量合并角色结果

这个节点是本工作流的关键。它接收历史暂存结果和当前章节结果，输出新的完整暂存结果。

系统提示词：

```text
你是小说角色表增量合并专家。你需要把“历史角色状态”和“当前章节角色分析”合并成新的角色状态。

合并规则：
1. 角色去重：同一角色的本名、外号、尊称、身份称号可以合并，但必须有文本证据或上下文强关联。不能仅凭同姓、同职业、同称号合并。
2. 角色定位优先级：核心角色 > 重要角色 > 边缘角色。
3. 定位升级：如果当前章节提供了更强证据，可以把角色从边缘升级到重要，或从重要升级到核心。
4. 定位降级：通常不要降级，除非历史状态明显来自解析错误。
5. 判断依据要累积，但每个角色最多保留 5 条最强依据，优先保留能体现主线推动、关键关系、关键行动、反复出现的证据。
6. 对角色名进行规范化：name 使用最明确、最稳定的称呼；其他称呼放入 aliases。
7. 输出必须是合法 JSON，不要输出 Markdown，不要输出解释性文字。

历史角色状态 JSON 结构：
[
  {
    "name": "标准角色名",
    "aliases": ["别名"],
    "position": "核心角色 | 重要角色 | 边缘角色",
    "evidence": [
      {
        "chapter": 1,
        "chapter_title": "章节标题",
        "text": "依据",
        "reason": "原因"
      }
    ],
    "chapters_seen": [1]
  }
]

请输出同样结构的 JSON 数组。
```

用户提示词：

```text
历史角色状态：
{{#Loop.role_state#}}

当前章节角色分析：
{{#解析单章 JSON.chapter_roles_json#}}
```

### 10. 代码：规范化合并结果

节点类型：代码节点  
语言：Python 3

输入：

| 变量 | 来源 |
| --- | --- |
| `text` | LLM：增量合并角色结果.text |
| `fallback_state` | Loop.role_state |

输出：

| 变量 | 类型 |
| --- | --- |
| `new_role_state` | string |

代码：

```python
def main(text: str, fallback_state: str):
    import json
    import re

    allowed = {"核心角色", "重要角色", "边缘角色"}

    def parse_json(value):
        raw = (value or "").strip()
        raw = re.sub(r"^```(?:json)?", "", raw).strip()
        raw = re.sub(r"```$", "", raw).strip()
        return json.loads(raw)

    try:
        data = parse_json(text)
    except Exception:
        try:
            data = parse_json(fallback_state)
        except Exception:
            data = []

    if not isinstance(data, list):
        data = []

    normalized = []
    seen = set()
    for item in data:
        if not isinstance(item, dict):
            continue

        name = str(item.get("name", "")).strip()
        if not name or name in seen:
            continue
        seen.add(name)

        position = str(item.get("position", "边缘角色")).strip()
        if position not in allowed:
            position = "边缘角色"

        aliases = item.get("aliases", [])
        if not isinstance(aliases, list):
            aliases = []
        aliases = [str(x).strip() for x in aliases if str(x).strip() and str(x).strip() != name]

        evidence = item.get("evidence", [])
        if not isinstance(evidence, list):
            evidence = []
        evidence = evidence[:5]

        chapters_seen = item.get("chapters_seen", [])
        if not isinstance(chapters_seen, list):
            chapters_seen = []
        chapters_seen = sorted({int(x) for x in chapters_seen if str(x).isdigit()})

        normalized.append({
            "name": name,
            "aliases": aliases[:10],
            "position": position,
            "evidence": evidence,
            "chapters_seen": chapters_seen,
        })

    rank = {"核心角色": 0, "重要角色": 1, "边缘角色": 2}
    normalized.sort(key=lambda x: (rank.get(x["position"], 9), x["name"]))

    return {
        "new_role_state": json.dumps(normalized, ensure_ascii=False),
    }
```

### 12. 最终格式化输出

节点类型：代码节点  
语言：Python 3

输入：

| 变量 | 来源 |
| --- | --- |
| `role_state` | Loop.role_state |

输出：

| 变量 | 类型 |
| --- | --- |
| `result` | string |

代码：

```python
def main(role_state: str):
    import json

    try:
        data = json.loads(role_state or "[]")
    except Exception:
        data = []

    result = []
    for item in data:
        if not isinstance(item, dict):
            continue

        evidence_texts = []
        for ev in item.get("evidence", []):
            if isinstance(ev, dict):
                chapter = ev.get("chapter")
                text = ev.get("text") or ""
                reason = ev.get("reason") or ""
                prefix = f"第{chapter}章：" if chapter else ""
                merged = f"{prefix}{text}"
                if reason:
                    merged += f"（{reason}）"
                if merged.strip():
                    evidence_texts.append(merged.strip())
            elif isinstance(ev, str) and ev.strip():
                evidence_texts.append(ev.strip())

        result.append({
            "角色名": item.get("name", ""),
            "角色定位": item.get("position", "边缘角色"),
            "判断依据": evidence_texts[:5],
        })

    rank = {"核心角色": 0, "重要角色": 1, "边缘角色": 2}
    result.sort(key=lambda x: (rank.get(x["角色定位"], 9), x["角色名"]))

    return {
        "result": json.dumps(result, ensure_ascii=False, indent=2),
    }
```

## 为什么不用原来的 Iteration

原工作流使用 Iteration，适合“每个章节互不依赖”的批处理。但这个新需求要求每一章都基于上一章的暂存结果继续合并，属于状态递进任务。Dify 的 Loop 节点更适合这种场景，因为它支持循环变量在多轮执行中持续存在并累积。

## 设计注意事项

1. 单章分析只负责识别本章角色，不做跨章节最终判断。
2. 增量合并节点负责跨章节去重、定位升级、依据筛选。
3. 每个角色最多保留 5 条依据，避免处理 100 章时变量过大。
4. 章节范围超过 100 时直接返回错误，不自动截断，避免用户误以为完整分析已完成。
5. 如果 LLM 单章输出 JSON 解析失败，该章会返回空角色列表，工作流继续使用上一轮 `role_state`，避免整条链路中断。
